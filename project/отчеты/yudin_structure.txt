Тема разработки
Сайт автосалона и технического обслуживания автомобилей.
Краткое описание объекта разработки
Разрабатывается веб-приложение, предназначенное для онлайн-продажи автомобилей и записи клиентов на техническое обслуживание.
Пользователь может просматривать каталог автомобилей, фильтровать модели по параметрам, оформлять покупку или оставлять заявку.
Кроме того, реализована возможность онлайн-записи на ТО, выбора типа услуги и даты визита.
Администраторы управляют каталогом, заказами и расписанием сервисного центра.
Выделение контекстов
Каталог автомобилей — хранение и отображение моделей, характеристик и цен.
Сценарий использования: Работа с каталогом автомобилей
Название: Просмотр и управление каталогом автомобилей на сайте автосалона Основной актер: Пользователь (клиент/покупатель)
Второстепенные актеры: Администратор (сотрудник автосалона)
Предусловия: Пользователь зашёл на сайт автосалона в раздел "Каталог автомобилей".
Постусловия: Пользователь получил нужную информацию о автомобилях; администратор поддерживает актуальность данных каталога.
Основные сценарии использования:
Сценарий 1: Пользователь просматривает каталог авто (для клиента) Поток событий:
Пользователь заходит в раздел "Каталог автомобилей".
Система отображает список доступных автомобилей с основными параметрами:
Фотография
Марка и модель
Год выпуска
Базовая цена
Статус (в наличии, под заказ, продан)
Пользователь может:
Листать каталог (пагинация или бесконечная прокрутка)
Сортировать по цене, году, названию
Переходить на детальную карточку автомобиля
Сценарий 2: Пользователь ищет автомобиль по фильтрам (для клиента) Поток событий:
Пользователь открывает панель фильтров.
Задаёт критерии поиска:
Марка/модель (выбор из списка)
Ценовой диапазон (от/до)
Год выпуска
Тип кузова (седан, внедорожник, хэтчбек и т.д.)
Двигатель (бензин, дизель, электро)
Коробка передач (автомат, механика)
Цвет
Статус (только в наличии)
Система обновляет список в реальном времени.
Пользователь видит только автомобили, соответствующие фильтрам.
Сценарий 3: Пользователь просматривает карточку модели (для клиента) Поток событий:
Пользователь кликает на автомобиль в общем списке.
Система открывает детальную карточку с информацией:
Галерея фотографий (внешность, интерьер, детали)
Полные технические характеристики:
Двигатель (объём, мощность, расход)
Размеры
Комплектации и доступные опции
Подробное описание
Цена для каждой комплектации
Доступные цвета с образцами
Сроки поставки (если под заказ)
Дополнительные возможности:
Сравнение с другими моделями
Сохранение в "избранное"
Запрос на тест-драйв
Кнопка "Купить" или "Оставить заявку"
Сценарий 4: Администратор добавляет/редактирует каталог (для сотрудника) Поток событий:
Администратор входит в административную панель сайта.
Переходит в раздел "Управление каталогом".
Добавление нового автомобиля:
Заполняет форму с полями:
Основные данные (марка, модель, год)
Технические характеристики
Описание
Цены для разных комплектаций
Загружает фотографии
Устанавливает статус (в наличии, под заказ)
Сохраняет запись
Редактирование существующего автомобиля:
Находит автомобиль в списке
Изменяет нужные данные:
Обновление цены
Изменение статуса (например, "продан")
Корректировка описания
Добавление новых фотографий
Сохраняет изменения
Удаление автомобиля:
Удаляет снятые с продажи модели
Или скрывает из публичного доступа (архивирует) Альтернативные потоки:
A1: Автомобиль временно недоступен
Система показывает статус "Под заказ" или "Ожидается поставка".
Предлагает оставить заявку на уведомление о поступлении. A2: Не найдено автомобилей по фильтрам
Система показывает сообщение "По вашему запросу ничего не найдено".
Предлагает:
Сбросить фильтры
Расширить критерии поиска
Оставить заявку на подбор
A3: Ошибка при редактировании администратором
Система проверяет обязательные поля (цена, модель).
При ошибке показывает конкретное сообщение (например, "Цена не может быть пустой").
Данные не сохраняются до исправления ошибок. Требования к системе:
Удобный интерфейс фильтрации с сохранением выбранных параметров.
Адаптивный дизайн для просмотра на мобильных устройствах.
Административная панель с разграничением прав доступа.
Поддержка загрузки нескольких фотографий для каждого автомобиля.
Интеграция с системой продаж (обновление статуса "продан" автоматически).
Бизнес-правила:
Цены отображаются с учётом действующих акций и скидок.
Автомобили со статусом "продан" остаются в каталоге с соответствующей пометкой (или скрываются по настройкам).
Изменения администратора в каталоге логируются (кто, когда, что изменил).
При изменении цены система может отправлять уведомления пользователям, добавившим автомобиль в "избранное".
Продажа автомобилей — оформление заказов, расчёт стоимости и статусов.

Запись на ТО — регистрация заявок клиентов на обслуживание.
Сценарий использования: Запись на техническое обслуживание автомобиля Название: Запись на ТО через сайт автосалона
Основной актор: Клиент (владелец автомобиля) Второстепенные акторы: Сотрудник сервиса (приёмщик/мастер)
Предусловия: Клиент зашёл на сайт автосалона в раздел "Техническое обслуживание". Постусловия: Создана заявка на ТО, клиент получил подтверждение.
Основной поток событий:
Клиент просматривает доступные услуги ТО (UC1):
Система отображает список услуг:
Плановое ТО (замена масла, фильтров)
Диагностика
Ремонтные работы
Замена запчастей
Для каждой услуги указана ориентировочная стоимость и время выполнения.
Клиент выбирает услугу и переходит к записи (UC2):
Клиент выбирает нужную услугу.
Заполняет форму:
Данные автомобиля (марка, модель, год, VIN)
Текущий пробег
Контактные данные (имя, телефон, email)
Желаемая дата и время визита
Система рассчитывает стоимость (UC3):
На основе выбранной услуги, модели авто и пробега система рассчитывает примерную стоимость.
Клиент видит итоговую сумму и подтверждает заявку.
Система обрабатывает заявку:
Назначает номер заявки.
Отправляет клиенту подтверждение на email/SMS.
Заявка попадает в систему управления для сотрудника сервиса.
Сотрудник сервиса управляет заявкой (UC5):
Видит новую заявку в панели управления.
Проверяет возможность выполнения в выбранное время.
Подтверждает или переносит запись (при необходимости связывается с клиентом).
Назначает мастера/бокс.
Заказывает необходимые запчасти (при необходимости).
Клиент отслеживает статус (UC4):
Через личный кабинет видит статус заявки:
"Принята"
"Подтверждена"
"В работе"
"Готово"
Получает уведомления об изменении статуса. Альтернативные потоки:
A1: Клиент не знает точную услугу
Клиент выбирает "Диагностику" как первоначальную услугу.
После диагностики сотрудник сервиса уточняет необходимые работы. A2: Нет свободных слотов на выбранное время
Система предлагает ближайшие доступные даты.
Клиент выбирает другое время или оставляет заявку в лист ожидания. A3: Необходимы запчасти, которых нет в наличии
Сотрудник сервиса информирует клиента о задержке.
Запрашивает согласие на заказ запчастей.
Переносит дату ТО. Требования к системе:
Календарь с доступными слотами для записи.
Интеграция с CRM для управления заявками.
Уведомления по email/SMS.
Личный кабинет клиента для отслеживания статуса.
Панель управления для сотрудников сервиса.
Бизнес-правила:
Запись возможна минимум за 24 часа до визита.
Предоплата не требуется при записи на стандартное ТО.
Клиент может отменить или перенести запись за 12 часов до визита без штрафа.
При повторном ТО в этом же сервисе - скидка 5% на работы.

ER-диаграмма: Каталог автомобилей
Назначение:
Эта ER-диаграмма описывает структуру данных для хранения и управления каталогом автомобилей на сайте автосалона. Она охватывает все аспекты
представления автомобилей: от производителей до конкретных комплектаций с фотографиями и опциями.
Ключевые сущности:
Основные сущности:
manufacturers - производители автомобилей
Хранит информацию о брендах: название, страна, логотип Связь: один производитель → много моделей (1:N)
car_models - модели автомобилей
Содержит технические характеристики: тип кузова, двигатель, коробка передач
Связь: одна модель → много комплектаций (1:N)
car_equipment - комплектации автомобилей
Определяет конкретные версии моделей с ценами и статусами наличия Связи: имеет фото, включает опции, доступна в цветах
Вспомогательные сущности:
car_colors - цвета автомобилей
Каталог доступных цветов с кодами и дополнительной стоимостью
equipment_options - опции комплектации
Дополнительное оборудование (климат-контроль, мультимедиа и т.д.)
car_photos - фотографии автомобилей
Галерея изображений для каждой комплектации
Связи и отношения:
manufacturers → car_models (1:N) - производитель выпускает несколько моделей
car_models → car_equipment (1:N) - модель имеет несколько комплектаций
car_equipment → car_photos (1:N) - комплектация может иметь несколько фото
car_equipment ↔ equipment_options (M:N через car_equipment_options) - комплектации включают опции
car_equipment ↔ car_colors (M:N через car_available_colors) - комплектации доступны в разных цветах
Особенности проектирования:
Использование ENUM для фиксированных значений (тип кузова, тип двигателя)
Десятичные типы для хранения цен и технических характеристик Отдельная таблица для связи M:N между комплектациями и опциями Поле stock_status для упра...
Использование в системе:
Отображение каталога автомобилей на сайте Фильтрация по характеристикам
Просмотр детальных карточек автомобилей

Продажа автомобилей:
ER-диаграмма: Продажа автомобилей
Назначение:
Эта ER-диаграмма описывает структуру данных для процесса продажи автомобилей: от оформления заказа до заключения договора. Она охватывает весь жизненн...
Ключевые сущности:
Основные сущности:
customers - покупатели
Хранит персональные и контактные данные клиентов Используется как в продажах, так и в ТО
orders - заказы
Центральная сущность процесса продажи
Содержит всю информацию о заказе: цены, статусы, способ оплаты Связи: связан с покупателем, комплектацией, цветом
payments - платежи
Хранит информацию об оплатах заказов
Поддерживает различные способы оплаты и статусы транзакций
Вспомогательные сущности:
order_selected_options - выбранные опции для заказа Связывает заказы с конкретными опциями комплектации
order_status_history - история статусов заказа Отслеживает изменения статусов заказа с комментариями
contracts - договоры
Хранит информацию о юридических документах сделки
managers - менеджеры по продажам
Сотрудники, ответственные за обработку заказов
Связи и отношения:
customers → orders (1:N) - клиент оформляет несколько заказов

orders → order_status_history (1:N) - заказ имеет историю изменения статусов orders → contracts (1:1) - каждый заказ сопровождается договором

ER-диаграмма: Запись на ТО
Назначение:
Эта ER-диаграмма описывает структуру данных для системы записи на техническое обслуживание автомобилей. Она охватывает весь процесс от создания заявки...
Ключевые сущности:
Основные сущности:
customers - клиенты сервиса
Та же сущность, что и в продажах (общая для всей системы) Обеспечивает единый профиль клиента
service_appointments - записи на ТО
Центральная сущность для сервисного обслуживания Содержит информацию о автомобиле, услуге, дате и времени
service_types - типы услуг ТО
Каталог доступных услуг с ценами и описаниями
Вспомогательные сущности: service_employees - сотрудники сервиса Мастера и приемщики, выполняющие работы
service_schedule - расписание работы сервиса Определяет доступные слоты для записи
service_parts - запчасти
Каталог деталей с информацией о наличии и совместимости
appointment_assignments - назначения на записи Связывает записи с сотрудниками сервиса
appointment_parts - запчасти для записи Связывает записи с требуемыми запчастями
Связи и отношения:
customers → service_appointments (1:N) - клиент может иметь несколько записей
service_appointments → appointment_assignments (1:N) - запись может быть назначена нескольким сотрудникам
service_employees → appointment_assignments (1:N) - сотрудник может работать по нескольким записям
service_appointments ↔ service_parts (M:N через appointment_parts) - запись требует запчастей
service_employees → service_schedule (1:N) - сотрудник имеет расписание работы
Особенности проектирования:
Отдельная сущность для типов услуг с категориями и ориентировочной длительностью
Система расписания с управлением доступными слотами Учет запчастей с проверкой наличия и совместимости
История статусов для отслеживания прогресса выполнения работ
Использование в системе:
Онлайн-запись на ТО с выбором даты и времени Просмотр доступных услуг и их стоимости Отслеживание статуса выполнения работ Управление расписанием серв...
Учет запчастей и их доступности
Общие принципы проектирования
Согласованность между контекстами:
Единый клиент - таблица customers используется и в продажах, и в ТО
Общие справочники - таблица car_models используется в каталоге и ТО
Согласованные идентификаторы - использование внешних ключей для связи контекстов
Нормализация:
Все диаграммы приведены к 3НФ (третьей нормальной форме) Разделение данных для устранения избыточности Использование связующих таблиц для отношений M:...
Расширяемость:
ENUM типы позволяют легко добавлять новые значения Отдельные таблицы для истории изменений
Поддержка дополнительных атрибутов через текстовые поля (notes, description)
Производительность:
Индексы на первичных и внешних ключах Разделение часто и редко изменяемых данных Архитектура, поддерживающая высокую нагрузку
Эти ER-диаграммы формируют полную и согласованную модель данных для системы автосалона, покрывая все бизнес-процессы, описанные в требованиях.
